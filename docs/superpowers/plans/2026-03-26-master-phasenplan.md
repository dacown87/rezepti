### Phase 2: Cook + Mobile — "Am Herd steht die App im Mittelpunkt"
**Ziel:** Fullscreen Cook Mode + PWA für mobile Nutzung.
**CC-Aufwand:** ~1-2 Sessions (~45-60 Min)
**Abhängigkeiten:** Phase 1 (stabile Basis)

#### Aufgaben:
- [ ] **PWA Setup**
  - Library: `vite-plugin-pwa` (Standard für Vite — generiert Manifest + Workbox-Service-Worker automatisch, kein manueller SW)
  - Workbox-Config: Cache-First für statische Assets; `NetworkOnly` für `/api/v1/*`
  - `public/manifest.json` + Meta-Tags in `index.html` + iOS `apple-touch-icon`
  - **Gelöst: HTTPS für LAN-Mobile** — zwei Stufen:
    1. **Graceful Degradation (immer):** Im Cook Mode prüfen ob `navigator.wakeLock` verfügbar ist; wenn nicht → UI-Hinweis "Bildschirm bleibt ohne HTTPS möglicherweise nicht an" statt stiller Fehler
    2. **Optional: Caddy als Docker-Service** mit `--profile https` in `docker-compose.yml` — Caddy generiert automatisch selbstsigniertes Zertifikat, kein `mkcert` nötig; ermöglicht PWA-Install + wakeLock von LAN-IP aus
    - Auf `localhost` funktioniert alles ohne HTTPS (Browser-Exception greift)
- [ ] **Fullscreen Cook Mode**
  - Neuer Route `/recipe/:id/cook`
  - Einzelner Schritt groß angezeigt (Swipe / Tap-Navigation)
  - Zutaten-Checkliste (abhakbar) in einem Side-Panel oder Drawer
  - Bildschirm-Wach-Lock (`navigator.wakeLock.request('screen')`)
  - Fortschrittsanzeige: "Schritt 3 von 7"
  - Zurück-Button zur RecipeDetail

**Tests:**
- PWA-Score in Lighthouse ≥ 80
- wakeLock API vorhanden (Graceful Degradation wenn nicht)
- Cook Mode: Schritt-Navigation vorwärts/rückwärts

### Phase 3b: Foto-Import — "Foto aus Buch oder Kamera"
**Ziel:** Rezept fotografieren → KI extrahiert Rezept (Vision-Modell bereits vorhanden).
**CC-Aufwand:** ~1 Session (~30 Min)
**Abhängigkeiten:** Phase 1

#### Aufgaben:
- [ ] **Foto-Import**
  - Neuer Tab im Extraction UI: "Foto hochladen"
  - `<input type="file" accept="image/jpeg,image/png,image/webp" capture="environment">` für mobile Kamera
  - Backend: `POST /api/v1/extract/photo` — MIME-Typ + max 10 MB Größe validieren vor Weiterleitung
  - Neuer Fetcher: `src/fetchers/photo.ts` → ruft `extractRecipeFromImage()` aus `src/processors/llm.ts` auf (bereits vorhanden!)
  - **Gelöst: Base64-Encoding in `photo.ts`** — `extractRecipeFromImage()` erwartet `imageUrl: string`; Groq-Vision-API unterstützt `data:image/jpeg;base64,...` direkt:
    ```ts
    const buffer = await file.arrayBuffer()
    const base64 = Buffer.from(buffer).toString('base64')
    const dataUrl = `data:${file.mimetype};base64,${base64}`
    await extractRecipeFromImage(dataUrl)
    ```
    Kein Temp-File, kein Cleanup nötig.
  - Pipeline-Integration: photo → vision → extraction → save (denselben Job-Manager-Flow wie URL-Import)

**Tests:**
- Unit: MIME-Validation (jpeg/png/webp ok; pdf → reject)
- Unit: Größen-Validation (10 MB ok; 11 MB → reject)
- Unit: Mock-Vision-Response → RecipeData wird korrekt geparsed
- Unit: Leeres Bild / kein Rezept erkannt → sinnvolle Fehlermeldung

### Phase 3c: Zutaten-Dictionary + Einkaufsliste — "Einkaufen und Kochen planen"
**Ziel:** Zutaten-Normalisierung + vollständige Einkaufsliste.
**CC-Aufwand:** ~2 Sessions (~60-80 Min)
**Abhängigkeiten:** Phase 3a + 3b (stabile DB + Photo-Pipeline)

#### Aufgaben:
- [ ] **Zutaten-Dictionary**
  - Neue DB-Tabelle: `ingredient_dictionary` — `canonical_name TEXT PRIMARY KEY`, `aliases TEXT` (JSON-Array)
  - Flag in `recipes`: `dictionary_checked INTEGER DEFAULT 0` (Migration via try/catch)
  - **Wann geprüft:** NICHT beim Import — beim Öffnen der Einkaufsliste ("Diese Zutaten klingen ähnlich — zusammenführen?")
  - **Gelöst: Ingredient-String-Parser** als `extractIngredientName()` in `src/ingredient-dictionary.ts`:
    - Baut auf `splitIngredient()` aus `scaling.ts` auf (trennt Zahl vom Rest)
    - Zweiter Unit-Regex auf `rest`: `/(g|kg|ml|l|EL|TL|Prise|Stk|Stück|...)\s+(.*)/i` → liefert den Namen
    - Randfälle ohne Zahl ("Salz") → ganzer String ist der Name
    - Beispiele: "200g Mehl"→"Mehl", "1 EL Olivenöl"→"Olivenöl", "Salz"→"Salz", "Saft von 1 Zitrone"→"von 1 Zitrone" (akzeptabler Randfallwert, Groq-Fallback klärt Ähnlichkeit)
    - Nur der extrahierte Name geht ins Levenshtein/Groq-Matching
  - **Matching-Strategie (Hybrid) — auf extrahierten `name` anwenden:**
    1. Levenshtein-Distanz ≤ `max(1, floor(0.3 × max(len(a), len(b))))` → direkt als "ähnlich" markiert (relative Schwelle — verhindert "Salz"↔"Malz" false positives)
    2. Grenzfälle → Groq-Call: "Ist '{new}' dasselbe wie '{canonical}'?" — alle Grenzfälle **parallel** (`Promise.all`) statt sequentiell
    3. Keine Ähnlichkeit → neue Zutat ins Dictionary aufnehmen
  - User-UI in Einkaufsliste: "Zusammenführen?" Modal mit Accept/Reject
  - `src/ingredient-dictionary.ts` — neues Modul
- [ ] **Einkaufsliste**
  - Neuer Tab `/shopping`
  - "Zur Einkaufsliste hinzufügen"-Button in RecipeDetail
  - Multi-Rezept-Unterstützung: Zutaten aus mehreren Rezepten zusammenführen
  - Aggregation nutzt **kanonische Namen aus Dictionary**
  - Abhak-Funktion (pro Item)
  - Export als Text/Clipboard
  - **Persistenz: SQLite-Tabelle `shopping_list`** (direkt — kein localStorage, da Phase 5 Wochenplan darauf zugreift)
  - API: `GET/POST/DELETE /api/v1/shopping` — neuer Router

**Tests:**
- Unit: `ingredient-dictionary.ts` — alle 7 Pfade: kein Match, Lev≤threshold, Grenzfall-Groq-true, Grenzfall-Groq-false, Groq-Timeout (→ kein Merge), Relative Threshold (kurze Wörter), Neue Zutat hinzufügen
- Unit: Mengen-Aggregation — gleiche Zutat summiert, unterschiedliche Zutaten angehängt, Dictionary-Lookup applied
- Unit: Shopping-API — CRUD Roundtrip
- Zutaten-Suche (Phase 4 Vorbereitung): SQL via Drizzle ORM parameterisiert — kein String-Concat

### Phase 4: Share + Smart — "Teilen und finden"
**Ziel:** PDF-Export, Zutaten-Suche in eigener Sammlung.
**CC-Aufwand:** ~1-2 Sessions (~60 Min)
**Abhängigkeiten:** Phase 3

#### Aufgaben:
- [ ] **PDF-Export / Rezeptkarte**
  - Library: `jsPDF` oder `react-pdf` (Frontend-seitig, kein Backend nötig)
  - Layout: Titel + Emoji + Zutaten + Schritte + QR-Code (→ source_url)
  - QR-Code: `qrcode.react` oder ähnliches
  - "Als PDF speichern"-Button in RecipeDetail
  - Optional: Druck-optimiertes CSS `@media print`
- [ ] **Rezept-Suche nach Zutaten**
  - Neues Eingabefeld in RecipeList oder separater `/search`-Tab
  - "Ich habe: Eier, Butter" → Filter der Sammlung nach Zutaten-Overlap
  - Backend: `GET /api/v1/recipes?ingredients=ei,butter` — OR-Logik: Rezept erscheint wenn es **mindestens eine** der Zutaten enthält
  - **Immer Drizzle ORM** für parameterisierte Queries (`.where(like(...))`) — kein String-Concat → SQL-Injection-sicher
  - Keine KI-Generierung — nur Suche in bestehender Sammlung

**Tests:**
- PDF-Generation: Prüfen ob Download ausgelöst wird
- Zutaten-Suche: API-Test mit bekannten Rezepten

### Phase 5: Growth — "Größer denken"
**Ziel:** Wochenplan, QR-Rezeptkarten, erste Schritte Richtung Erweiterbarkeit.
**CC-Aufwand:** ~2-3 Sessions (~120 Min)
**Abhängigkeiten:** Phase 3 (Einkaufsliste muss fertig sein)

#### Aufgaben:
- [ ] **Meal Planner / Wochenplan**
  - Neuer Tab `/planner`
  - 7-Tage-Kalenderansicht: Default "einfaches Zuweisen" (kein Drag & Drop — zu komplex auf Mobile); Drag & Drop optional Phase 6 mit `dnd-kit`
  - Scan-Feature: Physische Rezeptkarte fotografieren → Rezept wird im Planer eingetragen
    (nutzt Foto-Import aus Phase 3; QR-Scan optional nur Chromium via `BarcodeDetector` API)
  - "Einkaufsliste für diese Woche generieren" → aggregiert alle Rezepte
  - Persistenz: SQLite-Tabelle `meal_plan` (direkt — konsistent mit shopping_list)
- [ ] **QR-Code mit Rezept-JSON (offline)**
  - QR-Code enthält kompaktes Rezept-JSON, Base64-enkodiert
  - Limit: max 2 KB — primärer Wert ist der Fallback-URL-Fall; vollständiges Rezept im QR-Code ist Best-Effort
  - Fallback: Wenn Rezept > 2 KB → QR-Code enthält nur App-URL + Rezept-ID; User-Hinweis: "QR-Code enthält nur Link, kein vollständiges Rezept"
  - Lange Schritte werden auf 100 Zeichen getrimmt falls Offline-Einbettung gewünscht
  - Scanner-Flow: Kamera öffnen → QR scannen → Rezept importieren

**Tests:**
- Wochenplan: Persistenz-Test
- QR-Code-Roundtrip: Encode → Decode → Rezept korrekt

### Phase 6+: Multi-User (sehr späte Zukunft)
